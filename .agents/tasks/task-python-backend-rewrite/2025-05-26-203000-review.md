# Python/FastAPI backend rewrite replacing Node.js/Express

The entire backend has been rewritten from Node.js with Express, Prisma, and TypeScript to Python with FastAPI, SQLAlchemy, and synchronous SQLite. The rewrite preserves the same API routes (77 endpoints across 12 modules), JWT authentication scheme, role-based access control, and JSON response shape so the React frontend continues to work unchanged. The approach is a like-for-like port: same route prefixes, same request/response contracts, same HMAC-based QR attendance, same report export formats.

**Watch for:** Uploads directory served without authentication (confirmed) — the Node.js version required a valid JWT to access `/uploads/*` but the Python version mounts it as a public static directory. The online attendance duration check silently passes when lecture duration is null/0 (confirmed) — diverging from the Node.js behavior that defaults to 60 minutes. Exception swallowing in the global handler hides all errors in production (confirmed).

## High-level view

The auth and RBAC implementation is a faithful port: JWT with HS256 and 24h expiry, bcrypt password hashing, per-request database lookup to verify the user is still active.

File uploads lost their access control in the rewrite. The original Express app guarded `/uploads` behind the `authenticate` middleware. The Python version mounts it as an unprotected `StaticFiles` directory, meaning any unauthenticated request can fetch uploaded assignment submissions, resources, and student files by guessing or enumerating UUID filenames.

The online attendance path diverges from the original: when a lecture has no recorded duration (`null`), the Node.js code defaulted to 60 minutes and enforced the 70% threshold, while the Python code skips validation entirely (`if lecture_duration > 0` gate means null/0 duration = auto-pass). The QR attendance path also drops the 5-minute post-lecture window check that the Node.js version enforced.

Cascade deletion in admin routes manually deletes related records in application code rather than relying on database cascades. If a new relationship is added to a model, the delete handler will raise a foreign key constraint error unless manually updated.

The 27 tests cover auth, admin CRUD, and trainer lecture flows. Counsellor, student, attendance, evaluation, and report endpoints have zero test coverage — notable given the behavioral divergences in attendance logic.

<summary>Issues (7)</summary>

1. **Uploads served without auth** — The `/uploads` mount uses `StaticFiles` with no authentication. Add a custom route or middleware that verifies a JWT before serving files, matching the Node.js behavior.
2. **Online attendance skips validation for null-duration lectures** — When `lecture.duration` is `None` or `0`, the 70% threshold check is bypassed entirely. The Node.js code defaults to 60 minutes. Add a fallback default (e.g., 60) so the check always runs.
3. **Global exception handler swallows all errors** — The catch-all `Exception` handler returns a generic 500 with no logging. Add `logging.exception()` or similar so errors are visible in server logs.
4. **Delete handlers miss future FK relationships** — Admin delete routes manually cascade-delete related records. Consider adding `ON DELETE CASCADE` in SQLAlchemy model definitions.
5. **QR attendance window not enforced** — The Node.js code validates that QR attendance can only be marked within 5 minutes of lecture end. The Python port does not implement this check.
6. **Counsellor fee payment authorization gap** — If no counsellor profile is found (the `if counsellor:` guard), the code falls through without returning an error, allowing the payment to proceed without authorization verification.
7. **Docker Compose volume hides database** — The `db-data` volume mounts at `/app/backend`, overwriting the SQLite `dev.db` that was copied into the image. On rebuilds the volume's stale DB persists, potentially causing schema drift.

## Uploads directory auth regression

In the original Express server (`backend/src/index.ts`):

```javascript
app.use('/uploads', authenticate as any, express.static(path.join(__dirname, '../uploads')));
```

In the Python rewrite (`backend/app/main.py`):

```python
app.mount("/uploads", StaticFiles(directory=uploads_dir), name="uploads")
```

FastAPI's `StaticFiles` does not support dependency injection, so the authentication step was dropped. Uploaded files include assignment submissions, resource materials, and potentially sensitive documents. UUID-based filenames provide obscurity but not security — any authenticated user who receives a file path in an API response can share that URL publicly.

The fix requires either a custom route handler that streams files after checking the JWT, or a middleware wrapper around the static files mount.

## Online attendance duration divergence

The Node.js attendance service:

```javascript
const lectureDuration = lecture.duration || 60; // default 60 min
const threshold = lectureDuration * 0.7;
if (joinDuration < threshold) { throw new Error(...); }
```

The Python rewrite:

```python
lecture_duration = lecture.duration or 0
if lecture_duration > 0:
    required_duration = lecture_duration * 0.7
    if join_duration < required_duration:
        return error_response(...)
```

When `lecture.duration` is `None` (lecture hasn't ended yet, or duration wasn't recorded), the Python code silently allows attendance with zero join time. A student can mark online attendance for an in-progress lecture with `joinDuration: 0` and it succeeds. The Node.js code would require at least 42 minutes (70% of the 60-minute default).

## QR attendance window enforcement

The Node.js code (`attendance.service.ts`) explicitly checks that the current time is within 5 minutes of the lecture's end time:

```javascript
const windowEnd = new Date(lectureEnd.getTime() + ATTENDANCE_WINDOW_MINUTES * 60 * 1000);
if (now > windowEnd) { throw new Error('Attendance window has closed'); }
```

The Python port includes only the HMAC token age check (10 seconds), with a comment acknowledging the window check was simplified away. A student with a valid QR token could mark attendance hours after the lecture ended, as long as the token was generated within 10 seconds of their request.

## Counsellor payment authorization gap

In `routes/counsellor.py`, the `record_payment` endpoint:

```python
counsellor = db.query(Counsellor).filter(Counsellor.userId == current_user["userId"]).first()
if counsellor:
    cs = db.query(CounsellorStudent).filter(...).first()
    if not cs:
        return error_response("Access denied. Student not assigned to you.", 403)
```

If `counsellor` is `None` (the user has the COUNSELLOR role but no counsellor profile record), the entire authorization block is skipped and the payment proceeds. The `if counsellor:` should be a guard that returns a 404 when the profile is missing, consistent with how other counsellor endpoints handle this case.

## Error observability

```python
@app.exception_handler(Exception)
async def general_exception_handler(request: Request, exc: Exception):
    return error_response("Internal server error", 500)
```

No logging, no traceback, no context. Unhandled exceptions in route handlers will be invisible to the developer — the client sees a 500 but there's no server-side record of what failed.

## N+1 query patterns in counsellor and trainer routes

The counsellor `/students`, `/alerts`, and `/dashboard` endpoints iterate over students and issue per-student queries for attendance count and average marks. With 20 students this is manageable; with 200 it would produce ~400 database round-trips per request. The trainer `/batches/{id}/students` endpoint has the same pattern.

## Test coverage gaps

Not tested: counsellor endpoints, student endpoints, attendance (QR and online marking), evaluation/marks, mock interviews, assignments with file upload, resources, notifications/announcements, report generation and export, and the file upload utility. The absence of attendance tests is notable given the behavioral divergences identified above.
