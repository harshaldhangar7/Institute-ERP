# Institute ERP monorepo: full-stack greenfield implementation

A complete Institute ERP built as a Node.js/Express backend with Prisma ORM (SQLite), React/Vite frontend, and Docker containerization. The system implements four-role RBAC (Admin, Trainer, Counsellor, Student) with JWT authentication, covering attendance tracking (QR-based and online), evaluation, mock interviews, fee management, assignments with file uploads, reporting with PDF/Excel export, and notifications. The architecture uses a clean route-per-role pattern with Zod validation and service-layer separation.

**Watch for:** Hardcoded JWT secret falls through to production if env var is unset (confirmed). QR token comparison uses direct string equality rather than timing-safe comparison, enabling timing attacks on attendance tokens (confirmed). Deactivated users retain valid JWT tokens with no revocation mechanism (confirmed). Trainer endpoints lack ownership verification, allowing any trainer to end any other trainer's lecture or view any batch's students (confirmed). `DELETE` operations on students/trainers will fail with foreign key violations due to cascading relationship order (likely).

## High-level view

The backend follows a layered architecture: Express routes handle HTTP concerns and Zod validation, service modules encapsulate business logic and Prisma queries, and middleware gates authentication and role checks. Route groups are mounted per-role at the router level (`/api/admin`, `/api/trainer`, etc.) with a single `authenticate + roleGuard` middleware chain, which enforces coarse access control cleanly. Shared concerns like attendance and assignments are mounted with multi-role guards.

The security posture has a significant gap: JWT secrets use a hardcoded fallback that silently activates in production if the environment variable is missing, and there is no token revocation or session invalidation when accounts are deactivated. The QR attendance system implements HMAC-based tokens with a 10-second validity window, but the token comparison is vulnerable to timing attacks.

Trainer-scoped endpoints verify that the requester has the TRAINER role but do not verify that the trainer owns the resource they are modifying. A trainer can end another trainer's lecture or generate QR codes for lectures they don't teach, because the `generate-qr` endpoint only checks that the lecture exists, not who created it.

The data model uses SQLite through Prisma with UUID primary keys and appropriate unique constraints on junction tables. However, the `delete` operations delete the role-specific record first, then the user record, without a transaction -- and without deleting dependent records (attendances, marks, submissions), so deletions will fail on foreign key constraints in any non-trivial database state.

The frontend implements a standard React SPA pattern with AuthContext storing JWT in localStorage, axios interceptor for automatic token attachment, and a ProtectedRoute component that checks role before rendering. The 401 interceptor performs a hard redirect to `/login`, which works but loses application state.

<summary>Issues (9)</summary>

1. **Hardcoded JWT secret in production** -- The fallback `'super-secret-key-for-development'` activates silently if `JWT_SECRET` env var is unset. Remove the fallback and fail fast on startup if the variable is missing.
2. **QR token timing attack** -- `validateQRToken` uses `===` for HMAC comparison. Replace with `crypto.timingSafeEqual` to prevent timing-based token extraction.
3. **No token revocation on deactivation** -- Deactivating a user (`isActive = false`) has no effect on their existing JWT. Add an `isActive` check in the `authenticate` middleware or implement a token blacklist.
4. **Trainer cross-ownership on lectures** -- Any trainer can call `PUT /api/trainer/lectures/:id/end` or `POST /api/attendance/generate-qr` for lectures they do not own. Add `trainerId` ownership checks.
5. **Delete cascading failure** -- `deleteStudent` deletes the Student record then the User record, but does not delete dependent Attendances, Marks, Submissions, MockInterviews, or Fees first. These will throw foreign key errors. Wrap in a transaction with cascading deletes.
6. **Counsellor fee endpoint lacks ownership check** -- `POST /api/counsellor/fees/payment` accepts any `feeId` without verifying the fee belongs to a student assigned to the requesting counsellor.
7. **Pagination limit unbound** -- `parseInt(req.query.limit)` accepts any value. A client can pass `limit=100000` to dump the entire database. Cap the limit server-side.
8. **Static file directory traversal** -- Uploaded files are served via `express.static` on the `/uploads` path. The multer config uses UUIDs for filenames (which mitigates traversal), but there is no auth check on file access. Any unauthenticated user can access uploaded assignments and resources if they know the filename.
9. **Test database shared with dev** -- `setup.ts` sets `DATABASE_URL = 'file:./dev.db'`, so tests mutate the development database. Use a separate test database file.

## JWT secret and token lifecycle

The authentication middleware in `backend/src/middleware/auth.ts` derives the signing key from `process.env.JWT_SECRET || 'super-secret-key-for-development'`. The same pattern appears in `auth.service.ts`. If the environment variable is not set -- which can easily happen in a Docker deployment if the compose file is modified or a different orchestrator is used -- the system silently falls back to a publicly-known key. Any attacker can mint arbitrary tokens with any role.

The `loginUser` function checks `user.isActive` before issuing a token, but the `authenticate` middleware does not re-check this flag. Once a user has a valid token (24-hour expiry), deactivating them through the admin panel has no effect until the token naturally expires. For an ERP handling student data and fee records, this is a meaningful gap -- a disgruntled trainer retains full API access for up to 24 hours after termination.

```typescript
// auth.ts middleware - no isActive check after token verification
const decoded = jwt.verify(token, JWT_SECRET) as JwtPayload;
req.user = decoded;
next();
```

## QR attendance token verification

The `validateQRToken` function in `attendance.service.ts` computes an expected HMAC and compares it with `===`:

```typescript
return token === expectedToken;
```

Because HMAC outputs are hex strings of fixed length, a timing attack can iteratively determine correct characters by measuring response time differences. The 10-second token window makes exploitation harder but not impossible, especially in a local network (which is the likely deployment scenario for an institute). `crypto.timingSafeEqual(Buffer.from(token), Buffer.from(expectedToken))` is the fix.

The broader QR flow is sound: trainer generates a token scoped to a lecture, the token is HMAC-signed with a timestamp, and students must present it within 10 seconds. The 5-minute attendance window check after lecture end is also well-reasoned for the use case.

## Trainer resource ownership gap

The route structure gates trainers at the role level but not at the resource level. Consider this flow:

1. Trainer A creates a lecture for Batch X (their assigned batch).
2. Trainer B (assigned to Batch Y) can call `PUT /api/trainer/lectures/:id/end` with Trainer A's lecture ID.
3. The handler fetches the lecture, confirms it exists, and updates it. No check that `lecture.trainerId` matches the requesting trainer's ID.

The same pattern appears in `POST /api/attendance/generate-qr`. The `generate-qr` route is mounted under `/api/attendance` with `roleGuard(['TRAINER', 'STUDENT'])`, but the service function only verifies the lecture exists -- not that the requesting user is the trainer who owns it. A student could call this endpoint and generate QR codes for any lecture (the role guard allows students because the same router handles `mark-qr`).

```
POST /api/attendance/generate-qr
  -> roleGuard(['TRAINER', 'STUDENT'])  // students can hit this
  -> attendanceService.generateQR(lectureId)  // no user check
```

## Delete operations and referential integrity

The admin service deletes entities in this order:

```typescript
await prisma.student.delete({ where: { id } });
await prisma.user.delete({ where: { id: student.userId } });
```

But `Student` has foreign key relationships from `Attendance`, `Marks`, `MockInterview`, `Submission`, and `Fee`. Prisma with SQLite will reject the delete unless those are cleared first or the schema specifies `onDelete: Cascade` (it does not). The same issue exists for `deleteTrainer` (referenced by `Lecture`, `MockInterview`, `Assignment`, `Resource`, `TrainerBatch`) and `deleteBatch` (referenced by `Student`, `BatchModule`, `TrainerBatch`, `Lecture`, `Assignment`, `Announcement`).

Additionally, these two deletes are not wrapped in a `prisma.$transaction()`. If the student delete succeeds but the user delete fails (unlikely but possible), the database is left in an inconsistent state with an orphaned User record.

## Counsellor fee payment authorization

The `POST /api/counsellor/fees/payment` endpoint accepts `{ feeId, amount, method }` and calls `counsellorService.recordPayment`. The service fetches the fee record and updates it, but never verifies that the fee belongs to a student assigned to the requesting counsellor. Any counsellor can record payments against any student's fees. The counsellor's identity is looked up in other endpoints on this router but is not used here.

## Unbounded pagination

Admin list endpoints parse `limit` from the query string with no upper bound:

```typescript
const limit = parseInt(req.query.limit as string) || 10;
```

A request to `/api/admin/students?limit=999999` returns every student in one response. For an ERP that may eventually hold thousands of student records with nested includes, this is a DoS vector. A `Math.min(limit, 100)` cap would suffice.

## Upload file access without authentication

The express static middleware serves uploads at `/uploads` without any authentication gate:

```typescript
app.use('/uploads', express.static(path.join(__dirname, '../uploads')));
```

Since uploaded files include assignment submissions and resources (potentially containing proprietary course material or student work), unauthenticated access is a confidentiality concern. The UUID filenames provide obscurity but not security -- if a filename leaks in an API response to one user, any other user (or anonymous party) can retrieve the file.

## Announcement targeting logic

The `/api/notifications/announce` endpoint has a subtle targeting bug. When both `targetRole` and `batchId` are provided, the code builds two separate `where` clauses:

```typescript
if (parsed.data.targetRole) targetWhere.role = parsed.data.targetRole;
if (parsed.data.batchId) {
  const students = await prisma.student.findMany({...});
  targetWhere.id = { in: students.map(s => s.userId) };
}
```

If `targetRole` is set to `'TRAINER'` and `batchId` is also set, the final query becomes `{ role: 'TRAINER', id: { in: [student user IDs] } }`, which will match zero users (students don't have role TRAINER). The two filters are ANDed rather than ORed, making combined targeting silently broken.

## Frontend auth state and token expiry

The `AuthContext` initializes user state from `localStorage.getItem('user')` and parses it with `JSON.parse`. There is no validation that the stored token is still valid -- if the token has expired, the user sees the authenticated UI momentarily before the first API call triggers the 401 interceptor and redirects to login. This isn't a security issue (the backend rejects expired tokens), but it creates a flash of authenticated UI followed by an abrupt redirect.
