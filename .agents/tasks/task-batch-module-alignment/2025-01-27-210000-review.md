# Batch-module alignment across all roles

The change wires up batch-module relationships into admin serializers, adds a trainer dashboard endpoint, normalizes the student dashboard response field names, and adds counsellor alert data. On the frontend it surfaces module lists in admin/trainer batch views and hardens the shared Table component against non-array data. New pytest tests cover the trainer dashboard and the enriched admin serializers.

**Watch for:** joinedload with pagination produces duplicated rows when a batch has multiple students or modules (confirmed); the trainer dashboard has an N+1 lazy-load on `l.batch` for each upcoming lecture (confirmed); the student dashboard is a breaking API contract change with no versioning or frontend migration guard (confirmed).

## High-level view

The admin batch serializer now eagerly loads three relationship chains (modules, trainer, students) to populate new fields. This uses SQLAlchemy's `joinedload` against a query that also applies `.offset().limit()`, which inflates row counts before the ORM deduplicates — meaning the `total` count and per-page record count can drift apart from what the client expects.

The trainer dashboard is a straightforward aggregation endpoint. It queries `TrainerBatch`, then counts students and lectures, then fetches upcoming lectures and accesses `l.batch.name` without eager loading. With SQLite this triggers lazy loads inside the request; under connection pooling or async drivers it could fail silently or produce N+1 performance issues.

The student dashboard response changes field names (`moduleProgress` -> `modules`, `attendancePercentage` -> `attendance`, top-level `trainerName` moved into `batch.trainer`). This is a breaking change to any existing client. No API version header, no backward-compat shim, no coordinated frontend update to match — the frontend pages that already used the old names were reportedly already correct, meaning either the frontend was never actually working before this change, or there are untested pages that will break.

The counsellor alert computation iterates all assigned students and runs two queries per student (lecture count, avg marks). This is an O(n) query fan-out with no caching or batching.

<summary>Issues (7)</summary>

1. **Pagination row inflation** — `joinedload` on collections inflates result rows before `.offset().limit()`. Use `subqueryload` or apply `unique()` on the result to prevent batches appearing duplicated or pages returning fewer records than expected.
2. **Trainer dashboard N+1 on upcoming lectures** — Each lecture accesses `l.batch` lazily. Add `joinedload(Lecture.batch)` to the upcoming-lectures query to collapse into a single SQL round-trip.
3. **Breaking student dashboard contract** — Field renames (`moduleProgress`->`modules`, `attendancePercentage`->`attendance`, `trainerName` moved) break any client pinned to the old shape. Either ship both old and new field names temporarily, or verify that no other consumer depends on the old names.
4. **Counsellor alert O(n) query fan-out** — For each at-risk student, two separate queries run (lecture count, avg marks). Batch these into aggregate queries up front.
5. **Trainer Batches page data mismatch** — The frontend renders `batch.students?.length` and `batch.status` but the `/api/trainer/batches` endpoint returns neither `students`/`studentCount` nor `status` (only `isActive`). The page always shows "0 students" and an undefined status badge.
6. **`from datetime import date` inside function body** — Import at module top. The inner import shadows reasoning about dependencies and slightly slows repeated calls.
7. **Counsellor alerts missing severity/timestamp** — `recentAlerts` has no date field and no severity ordering, so the frontend cannot sort by recency or distinguish new from old alerts despite the "recent" naming.

## Pagination row inflation with joinedload

The admin `/batches` endpoint applies `joinedload` on three collection relationships simultaneously:

```python
batches = db.query(Batch).options(
    joinedload(Batch.batchModules).joinedload(BatchModule.module),
    joinedload(Batch.trainerBatches).joinedload(TrainerBatch.trainer).joinedload(Trainer.user),
    joinedload(Batch.students),
).offset(offset).limit(limit).all()
```

`joinedload` on collections works by adding LEFT OUTER JOINs to the SQL statement. When a single batch has 3 modules and 5 students, the database returns `3 * 5 = 15` rows for that one batch. SQLAlchemy's legacy `Query` API deduplicates entities in Python, but the `.limit(10)` applies to the raw SQL rows, not to the deduplicated entities. So if the first batch produces 15 joined rows, `LIMIT 10` might return only data for 1-2 actual batches instead of 10.

The `total` count is computed separately with `db.query(Batch).count()` and stays correct, so the client sees `total: 50` but only receives 2 items on page 1. This is a known SQLAlchemy pitfall. The fix is to use `subqueryload` (or `selectinload`) instead of `joinedload` for collection relationships when pagination is involved, or to paginate IDs first and then load relationships in a second query.

The same issue applies to the `/modules` endpoint, though with only one collection join the inflation is less severe.

## Trainer dashboard lazy-load N+1

The upcoming-lectures query fetches up to 5 `Lecture` objects, then the serialization loop accesses `l.batch.name`:

```python
upcoming_data = [{
    "topic": l.topics or "No topic",
    "date": l.date.isoformat() if l.date else None,
    "batch": l.batch.name if l.batch else None,
} for l in upcoming]
```

No `joinedload(Lecture.batch)` is applied to the query, so each `l.batch` triggers a separate SELECT. With SQLite in synchronous mode this works but costs 5 extra round-trips. Under an async session (if the app ever migrates to async SQLAlchemy) this would raise a `MissingGreenlet` error. Adding `.options(joinedload(Lecture.batch))` to the upcoming-lectures query eliminates the issue.

## Breaking student dashboard contract

The diff renames response fields:

```python
# Before
"attendancePercentage": ...,
"moduleProgress": [...],
"trainerName": ...,

# After
"attendance": ...,
"modules": [...],
# trainerName removed, now inside batch_info["trainer"]
```

The FEAT-002 findings state that the frontend "already had correct field mappings" — implying the frontend was already using the new names before this backend change shipped. That means either the frontend was broken against the old backend, or there's a version mismatch in the described findings. Either way, this is a hard breaking change with no deprecation path. Any mobile client, integration test, or third-party consumer using the old field names will silently receive `undefined`.

## Counsellor alert fan-out

The at-risk calculation iterates every student assigned to the counsellor and runs two queries per student: one `Lecture.count()` filtered by batch and one `func.avg(Marks.score)`. For a counsellor with 30 students this is 60 additional queries per dashboard load. The lecture counts could be pre-aggregated with a single grouped query on `batchId`, and marks averages can be computed with a single grouped query on `studentId`.

## Trainer Batches page data mismatch

The Trainer Batches page renders both `batch.status` (in a Badge) and `batch.students?.length`:

```tsx
<Badge variant={batch.status === 'ACTIVE' ? 'success' : 'default'}>{batch.status}</Badge>
<p className="text-sm text-gray-500">{batch.students?.length || 0} students</p>
```

But the `GET /api/trainer/batches` endpoint returns:

```python
data = [{
    "id": b.id, "name": b.name,
    "startDate": ..., "endDate": ..., "isActive": ...,
}]
```

Neither `students`/`studentCount` nor `status` is included. The Badge renders the text "undefined" and the student count is always 0. The admin batches serializer already computes both fields — the trainer batches endpoint should either reuse that serializer or derive `status` from `isActive` and include a student count.

## Counsellor alerts missing temporal data

The `recentAlerts` array is computed from the current at-risk snapshot with no timestamp or ordering signal:

```python
recent_alerts.append({
    "message": f"Student has {', '.join(alert_reasons).lower()}",
    "studentName": student.user.name if student.user else "Unknown",
    "type": alert_reasons[0],
})
```

The name "recentAlerts" implies time-ordered data, but the array is just the current set of at-risk students in iteration order. There's no `date` field for the frontend to display when the alert was generated, and no severity ranking. If the frontend expects to show "2 hours ago" or sort by urgency, this shape won't support it.

## Inner import in trainer dashboard

```python
from datetime import date
today = date.today()
```

This import lives inside the `dashboard` function body despite `from datetime import datetime` already being at the module top. Moving `from datetime import date` to the top-level import block is a minor hygiene fix.
